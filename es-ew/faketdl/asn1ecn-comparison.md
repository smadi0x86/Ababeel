# FakeTDL Modeled in ASN.1 with Encoding Control Notation

This was generated by ChatGPT5 given the FakeTDLSpecification.md on 2025-09-30. 
The prompting asked for it to create a minimally redundant ASN.1 + ECN, so it 
is using type definitions and annotating the type definitions in the ECN to 
avoid specifying the backbone structure of each message redundantly. 

## Abstract Types

This provides the abstract type definition and would normally be in a ".asn" file. 

```asn1

FakeTDL DEFINITIONS AUTOMATIC TAGS ::=
BEGIN

-- Atomic types (abstract constraints; encoding given in ECN)
MessageID      ::= INTEGER (0..99999)                 -- 0 = "no statement"
Boolean1       ::= INTEGER (0..1)
Course         ::= INTEGER (0..359)
Speed          ::= INTEGER (0..2000)
Elevation      ::= INTEGER (-1000..20000)             -- units: 25 ft (semantic)
Latitude       ::= REAL                                -- 32-bit IEEE-754 in ECN
Longitude      ::= REAL

TimeBCD        ::= OCTET STRING (SIZE(3))             -- HH,MM,SS (BCD) per spec
UnitNumber     ::= IA5String (SIZE(5))                -- A–Z, 0–9 (pattern per spec)
TrackNumber    ::= IA5String (SIZE(5))
DescriptionText ::= IA5String (SIZE(1..38))           -- allowed chars per spec

IFF ::= ENUMERATED { friend(0), enemy(1), unknown(2) }
EntityCategory ::= ENUMERATED { land(0), sea(1), sub(2), air(3), weapon(4) }

-- Common header for all messages
Header ::= SEQUENCE {
  messageType   IA5String (SIZE(1)),                  -- 'T','D','A'
  sourceUnit    UnitNumber,
  sendTime      TimeBCD
}

-- Body choices (no encodings here; rely on type encodings & defaults)
TrackMessage ::= SEQUENCE {
  ackRequired     Boolean1,
  messageID       MessageID,
  sourceLat       Latitude,
  sourceLon       Longitude,
  sourceElev      Elevation,
  trackNumber     TrackNumber,
  observationTime TimeBCD,
  obsLat          Latitude,
  obsLon          Longitude,
  obsElev         Elevation,
  pointType       IA5String (SIZE(1)),                -- 'S','E','W'
  estOrActual     IA5String (SIZE(1)),                -- 'E','A'
  course          Course,
  speed           Speed
}

IdentityMessage ::= SEQUENCE {
  ackRequired     Boolean1,
  messageID       MessageID,
  trackNumber     TrackNumber,
  observationTime TimeBCD,
  iff             IFF,                                -- F/E/U characters on wire
  category        EntityCategory,                     -- L/S/U/A/W on wire
  type            INTEGER (0..65535),
  description     DescriptionText
}

AckMessage ::= SEQUENCE {
  destUnit        UnitNumber,
  ackedIDs        SEQUENCE (SIZE(1..12)) OF MessageID
}

-- Top-level message with header + body
Message ::= SEQUENCE {
  header          Header,
  body            CHOICE { track TrackMessage, identity IdentityMessage, ack AckMessage }
}

-- Optional: explicit envelope type if you want to label padding behavior
MessageEnvelope ::= Message

END

```

## Encoding Control Notation

Now the ECN part, which would be in a ".ecn" file. 

```ecn
FakeTDL-ECN DEFINITIONS ::=
BEGIN

-- ===== Module-wide defaults (apply everywhere unless overridden) =====
DEFAULTS:
  INTEGER        -> BIG-ENDIAN, FIXED as needed by field widths
  IA5String(n)   -> EXACTLY n OCTETS (no terminator, no length prefix)
  REAL           -> IEEE-754 BINARY32 (BIG-ENDIAN byte order)
  ENUMERATED     -> IA5String(1) when the spec prescribes a single ASCII tag;
                    otherwise default to Unsigned minimal (override below)
  SEQUENCE/CHOICE order -> ASN.1 field order

-- ===== Per-type mappings (write once; reused everywhere) =====

-- 4-byte unsigned on the wire; abstract range 0..99999
MessageID        -> Unsigned(32)

-- Single-octet boolean
Boolean1         -> Unsigned(8)

-- Angles & speed are 2-byte unsigned
Course           -> Unsigned(16)
Speed            -> Unsigned(16)

-- Elevation is 2-byte signed (units: 25 ft, semantic)
Elevation        -> Signed(16)

-- Lat/Lon are 32-bit IEEE floats
Latitude         -> IEEE-FLOAT(32)
Longitude        -> IEEE-FLOAT(32)

-- Times are exactly 3 octets, BCD: HH,MM,SS
TimeBCD          -> OCTETS(3)  -- each octet is BCD of two digits

-- Unit/Track numbers are fixed 5 ASCII bytes
UnitNumber       -> IA5String(5)
TrackNumber      -> IA5String(5)

-- Identity/Category appear on the wire as 1 ASCII tag per spec
IFF              -> IA5String(1)        -- 'F','E','U'
EntityCategory   -> IA5String(1)        -- 'L','S','U','A','W'

-- Description is fixed-size field; pad with NUL if shorter
DescriptionText  -> IA5String(38) PAD WITH 0x00

-- ===== Top-level selection & framing =====

-- The CHOICE is selected by Header.messageType: 'T','D','A'
Message CHOICE-SELECTION:
  body BY header.messageType
    'T' -> track
    'D' -> identity
    'A' -> ack

-- Entire message is exactly 64 octets; pad with 'X' (0x53)
MessageEnvelope FRAMING:
  ENCODE Message
  PAD WITH 0x53 TO LENGTH 64

END


```

## Comparison Points for DFDL and ASN.1+ECN

Some of these points are based on the above ASN1+ECN in comparison to the DFDL Schema for 
FakeTDL. 
Others anticipate issues that have been encountered trying to model legacy data formats using DFDL. 

- ASN.1 is easier to read. More human friendly than DFDL's XML-Schema-based notation.
- ASN.1 ECN's keywords come from an entirely different naming tradition than DFDL's properties. 
- ASN.1 ECN has nothing like a calendar pattern language nor number pattern language.
  It is almost entirely about binary data types, not textual data (nor BCD - which is like text.)
- ASN.1 has no way to construct date/time from binary data such as the BCD representation used 
  by fakeTDL. You can't do it directly, and you can't do it using any sort of expression 
  calculations. 
  - ASN.1 does have parameterized types and simple expressions, but lacks constructors for 
 date/time types. 
  - Arguably, DFDL is missing many needed constructors also. They just are 
 less needed given the date/time pattern language.
  - ECN does have something like DFDL inputValueCalc and outputValueCalc properties. You can 
    specify a calculation to be used when encoding data that is distinct from those used to 
    decode the data.
    The limitation really seems to be the lack of constructors and such one can call. 
- Even if you have no interest in PER/DER/BER type encodings you have to separate the ECN from 
  the logical structure part. This is an intentional design choice for ASN.1. 
- Tricky things 
  like [looking ahead into the data to enform an earlier parse](
  https://github.com/OpenGridForum/DFDL/issues/66#issuecomment-2657507951)
  are not possible in ASN.1 it seems. 
  This comes up in legacy data. 
  DFDL/Daffodil does have some techniques like the `dfdlx:lookAhead` function to enable this. 
- ASN.1 ECN lacks an equivalent/alternative to the DFDL `bitOrder` property. 

ASN.1 + ECN does not seem to be designed to handle existing difficult data formats, but to allow 
new data formats to be expressed with a choice of pre-existing (BER/PER/DER/CER/XER/JER) or 
custom (ECN) encodings. But the ECN capability is not flexible enough to model legacy data formats. 

One could write a DFDL Schema to ASN.1 converter, but not a DFDL properties to ECN converter.
Daffodil could be extended with BER/PER/DER/etc. InfosetOutputter and InfosetInputter
implementations and that would then provide interoperability with ASN.1's standard encodings.
But one would not be able to take an arbitrary ASN.1 schema and create a corresponding DFDL
schema from it as there are type-system incompatibilities around object IDs in ASN.1 and other
incompatibilities like the ASN.1 ANY DEFINED BY construct.
These have no analog in DFDL.

## In Summary
- ASN.1 is like designing a data class and then picking a serializer (JSON, protobuf, DER, PER, 
ECN…). You start from the logical structure you want, then choose or customize a representation.
- DFDL is like writing a parser grammar that says, byte by byte (or bit by bit), how to
  turn this specific file or stream into structured data. You start from the physical data, and
  the logical structure you get sometimes necessarily reflects aspects of the 
  representation, as opposed to what you might prefer logically.
